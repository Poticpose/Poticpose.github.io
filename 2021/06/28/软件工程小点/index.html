软件工程基础

软件危机特征主要体现在7个方面

  • 在软件系统运行过程中，经常出现功能、性能等严重问题；
  • 软件产品的质量、可靠性和安全等方面，时常不能达到标准要求；
  • 软件开发管理差，对成本和进度难以估计；
  • 系统时常出现无法维护、升级或更新的问题；
  • 开发效率低，无法满足应用快速发展与更新升级的实际需要；
  • 研发成本难以控制，在总成本中所占的比例不断大幅上升；
  • 软件开发没有标准、完整、统一规范的文档资料。

软件危机产生的主要原因在6个方面

  • 软件开发的规模、复杂度和需求量都在不断增加；
  • 软件需求分析与设计不完善、有欠缺，致使软件开发、维护和管理或文档出现问题；
  • 没有按照工程化方式运作，开发过程无统一标准、准则和规范方法；
  • 研发人员之间或同用户相互交流沟通不够或文档资料不完备；
  • 软件测试调试不规范、不细致，提交的软件质量不达标；
  • 在软件运行过程中，忽视了正常有效的维护和管理；

解决软件危机的主要措施

  • 技术方法；
  • 开发工具；
  • 组织管理；

软件 = 程序 + 数据 + 文档 + 服务

软件工程的定义是：应用计算机科学、数学及管理科学等原理，开发软件的过程。软件工程借鉴传统工程的原则、方法以提高质量、降低成本。

软件生存周期是从开始研发软件到软件停止使用的整个过程。是指软件产品从用户提出开发需求开始，经过开发、使用和维护，直到最后淘汰的整个周期。

常用软件开发模型有：

  • 瀑布模型
  • 增量模型
  • 快速原型模型
  • 螺旋模型
  • 喷泉模型

可行性分析及研发策划

可行性分析可包括5个方面：

  • 技术可行性分析
  • 经济可行性分析
  • 社会可行性分析
  • 开发方案可行性分析
  • 运行可行性分析

练习

软件可行性分析的目的就是用最小的代价在尽可能短的时间内确定该软件项目是否能够开发，是否值得开发。

效益分析分为有形效益和无形效益两种。有形效益可以用货币的时间价值、投资回收期、纯收入等指标进行度量，无形效益主要从性质上、心理上进行衡量，很难直接进行量的比较。

研究开发资源的有效性是进行（技术）可行性分析的一方面。

研究软硬件资源的有效性是进行（技术可行性）研究的一方面。

软件需求分析

需求分析是软件开发过程中的一个非常重要的阶段，是整个软件开发的重要依据和基础，关系软件系统开发的成败，是决定产品质量的关键。

软件需求分：

  • 业务需求

  • 用户需求

  • 功能需求

  • 非功能需求

    软件需求分析的主要目的是获取用户及项目的具体需求，通过对实际需求的获取、分析、文档化和验证等需求分析过程，为软件的进一步设计和实现提供依据。

    数据字典有四类条目：数据流、数据项、文件及基本加工。

练习

用于描述基本加工说明的3种描述工具是结构化语言、判定表、判定树。

在需求分析阶段，需要确定问题的综合需求，这些需求包括功能需求、性能需求、环境需求、用户界面需求。另外还有可靠性、安全性、保密性、可移植性、可维护性等方面的功能。

在绘制分层的DFD时，父图与子图的输入输出数据流要平衡。

DFD是常用的进行软件需求分析的图形工具，其基本符号是（加工、数据流、数据存储和外部实体）

判定表和判定树是DFD中用以描述加工的工具，它通常描述的对象是（组合组件）

软件设计

软件设计也称为系统设计，是利用软件研发技术、工具和方法，确定新软件系统的具体物理实现方式、方法和方案的过程。

软件设计的总体目标是：将软件需求分析阶段确定的各种功能对应的逻辑模型，转换为具体的物理模型。

软件设计分为两大阶段：总体设计和详细设计。

总体设计的总原则和过程是：由宏观到微观、逐步求精的原则，定性与定量分析相结合、分解与协调相结合和模块化方法，并兼顾软件的通用性、关联性、整体性和层次性。

模块的内聚和耦合：

内聚表示一个模块内部各个组成元素之间相互结合得紧密程度。

耦合是指在软件模块之间互相依赖的方式和程度。

详细设计主要完成对软件模块的内部过程的具体设计和描述。

详细设计的工具：图形工具、表格工具、语言工具

练习

系统详细设计阶段最后产生的文档是详细设计说明书。

软件详细设计工具可分为图形工具、设计语言和表格工具。

为了提高模块的独立性，模块之间最好是（数据耦合）

详细设计的任务是确定每个模块的（算法和数据结构）

面向对象开发方法

对象是系统中的基本运行的实体，是代码和数据的集合。

对象具有三要素：对象标识、属性、服务。

类是对具有相同属性和服务的一组对象的抽象定义。

继承是父类与子类之间共享数据结构和方法的一种机制。

练习

面向对象的程序设计语言应具备面向对象方法所要求的4个成分，它们是类、对象、继承和消息通信。

只有类的共有界面的成员才能成为使用类的操作，这是软件设计的（信息隐蔽）原则。

当且仅当一个操作对类的实例的用户有用时，它才是类公共界面的一个成员，这是软件设计的（最小界面）原则

软件编程实现

软件实现是通过软件编程或利用复用与生成技术集成、测试和调试，具体完成软件产品的过程。

软件实现的主要目标是按照软件需求分析和详细设计的具体指标要求，选择合适的技术路线、工具、方法和编程语言。

软件实现的主要任务是将软件详细设计的结果转换为目标软件。

软件编程框架是可以复用的软件设计架构。

软件版本包括：

  • 中间版本
  • α版本
  • β版本
  • 发布版本
  • 维护版本

选取编程语言的标准：

理想化标准：

  • 有利于实现的优质高效
  • 有利于实现功能性能及可靠性
  • 有利于降低软件开发及维护成本

实用性标准：

  • 程序语言自身特性
  • 软件的应用领域
  • 软件开发环境
  • 软件开发方法
  • 编程人员的熟悉情况
  • 算法和数据结构的复杂性
  • 软件可移植性要求

软件测试与维护

软件测试的目的是：尽可能多地找到软件中的错误，而不是证明软件的正确性。

软件调试的方法：

  • 强行排错
  • 回溯法排错
  • 归纳法排错
  • 演绎法排错

练习

软件测试的目的是发现软件的错误。

单元测试一般以白盒测试为主，测试的依据是系统的模块功能规格说明

黑盒测试法是通过分析程序的功能来设计测试用例的方法。

成功的测试是指运行测试用例后（发现了程序的错误）。

软件的集成测试工作最好由（不属于该软件开发组的软件设计人员）

什么是白盒测试法？什么是黑盒测试法？

黑盒测试也称为功能测试或黑箱测试，其中的“黑盒”指测试人员只知道被测软件的界面和接口外部情况，不必考虑程序内部的逻辑结构和特性，只根据程序的需求分析规格说明，检查其功能是否符合。

白盒测试主要对程序内部结构执行路径的测试，也称为透明盒测试。测试人员将测试对象看作一个打开的盒子，搞清软件内部逻辑结构和执行路径后，利用其结构及有关信息设计测试用例，对程序的所有逻辑路径进行测试，以检测不同检查程序的实际状态与预期状态一致性。

软件调试一般经过哪些步骤，采取哪些方法？

步骤：重现问题、定位根源、确定解决方案、验证方案。

方法：强行排错、回溯法排错、归纳法排错、演绎法排错。

软件项目管理

软件项目的风险管理是指对软件项目可能出现的风险，进行识别、评估、预防、监控的过程。

风险估算也称为风险预测，主要从风险发生的可能性和风险发生所产生的后果两个方面评估风险。

练习

软件项目启动过程的主要任务是确定项目的目标、约束和自由度，并进行投资/效益分析和可行性研究，如果符合企业的商业目标，则确定立项。

软件项目估算的内容主要有软件规模估算、工作量估算、成本估算和进度估算。

软件项目开发团队都有哪些角色？怎样组织？

角色：项目经理、分析人员、架构师、开发人员、测试人员、部署人员

项目验收的条件是什么？项目验收要经历哪些过程？

过程：评审项目验收计划、项目团队验收自查、项目最终评审、项目验收、项目总结和资料归档。
