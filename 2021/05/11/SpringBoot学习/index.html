注解

  • @Configuration：告诉 Springboot 这是一个配置类
  • @SpingBootApplication：开启组件扫描和自动配置功能
  • @ComponentScan：启动组件扫描
  • @EnableAutoConfiguration：自动配置
  • @Bean：给容器中添加组件。以方法名为 id，返回类型就是组件类型。返回的值，就是组件在容器中的实例
  • @Controller：返回页面，显示用
  • @RestController：替代了 @ResponeseBody + @Controller 返回JSON格式数据
  • @ResponseBody：返回JSON数据
  • @ControllerAdvice：捕获全局异常
  • @ExceptionHandler(RuntimeException.class)：返回当前异常

文件上传与下载

上传文件必须将表单 method 设置为 POST，并将 enctype 设置为 multipart/formdata。

只有这样，浏览器才会把用户所选文件的二进制数据发送给服务器，SpringMVC 在文件上传时，会将上传的文件映射为 MultipartFile 对象，并对 MultipartFile 对象进行文件解析和保存。

MultipartFile 接口有以下几个常用方法：

  • byte[] getBytes()：获取文件数据。
  • String getContentType()：获取文件 MIME 类型，如 application/pdf、image/pdf 等
  • InputStream getInputStream()：获取文件流
  • String getOriginalFileName()：获取上传文件的原名称
  • long getSize()：获取文件的字节大小，单位为 Byte
  • boolean isEmpty()：判断是否有上传的文件
  • void transferTo(File dest)：将上传的文件保存到一个目标文件中

定时器

  • Timer：Java 自带的 java.util.Timer 类，这个类允许调度一个 java.util.TimerTask 任务。使用这种方式可以让程序按照某一个频率执行，但不能在指定时间运行
  • ScheduledExecutorService：jkd 自带的一个类，基于线程池设计的定时任务类，每个调度任务都会分配到线程池中的一个线程去执行，也就是说任务是并发执行的，它们之间互不影响
  • Spring Task：Spring 自带的 Task，可以看做一个轻量级的 Quartz
  • Quartz：功能强大的调度器，可以让程序在指定的时间执行，也可以按照某一个频率执行

简单的定时任务可以直接使用 Spring Task，复杂任务则通过 Quartz 实现

cronExperssion 表达式：

字段 允许值 允许的特殊字符
秒 0-59 , - * /
分 0-59 , - * /
小时 0-23 , - * /
日期 1-31 , - * / L W C
月份 1-12 , - * /
星期 1-7 , - * / L C #
年 1970-2099 , - * /

特殊字符说明：

  • “ * ”：代指所有值。
  • “ - ”：指定范围。
  • “ , ”：指定另外之，做分割
  • “ ? ”：制定非明确的值
  • “ L ”：指定某月或星期的最后一天
  • “ W ”：距离指定日期最近的一个星期日
  • “ # ”：在某月中第几个星期

表达式 意义
“0 0 12 * * ?” 每天中午 12 点触发
“0 15 10 * * ?” 每天上午 10:15 触发
“0 15 10 * * ?” 每天上午 10:15 触发
“0 15 10 * * ? *” 每天上午 10:15 触发
“0 15 10 * * ? 2005” 2005 年的每天上午 10:15 触发
“ 0 * 14 * * ?” 每天下午 2 点到下午 2:59 期间的每一分钟触发

热部署

devtools 热部署

热部署会监听到有 Class 文件改动，就会创建一个新的 ClassLoader 进行加载该文件，经过一系列过程，最终呈现结果。

深层原理是用两个 ClassLoader，一个 ClassLoader 加载那些不会改变的类（第三方 Jar 包），另一个 ClassLoader 加载会更改的类，称为 restart ClassLoader，这样在有代码更改的情况下，原来的 restart ClassLoader 被丢弃，重新创建一个，由于需要加载的类偏少，所以缩短了重启时间。

整合数据库

关系型数据库：

优点：

  • 容易理解：二维表结构贴近逻辑世界，关系模型更易理解
  • 使用方便：通用的SQL语言
  • 易于维护：丰富的完整性，降低数据冗余概率
  • 支持SQL，可用于复杂查询

缺点：

  • 为了维护一致性所付出的巨大代价——读写性能较差
  • 固定的表结构
  • 高并发读写需求
  • 海量数据的高效率读写较差

非关系型数据库：

优点：

  • 格式灵活：可以以键值对、文档、图片等形式存储
  • 速度快：NoSQL 可以使用硬盘或者随机存储器为载体
  • 高扩展性
  • 成本低

缺点：

  • 数据和数据之间没有关系
  • 没有关系，没有强大的事务保证数据的完整和安全

Redis

Redis 是基于内存的单线程高性能键值对型数据库，定期通过异步操作把数据库数据 flash 到硬盘进行保存。

Redis 默认支持 16 个数据库，这些数据库的默认命名都是从 0 开始递增的数字。

当我们连接到 Redis 时，默认操作的是 0 号数据库，可以通过 select 更换数据库，databases 修改默认数据库个数。

注意：

  1. Redis 不支持自定义数据库名
  2. Redis 不支持为每个数据库设置密码，默认数据库密码为空
  3. Redis 的多个数据库之间不完全隔离，flushall 会清空所有数据库的数据
  4. flushdb 会清空当前数据库的数据
